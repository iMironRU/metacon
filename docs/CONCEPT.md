# Тренажёр-конструктор задач «1С:Специалист» — Концепция

`v0.1.0` · 2026-06-16 · статус: проектирование (диспетчер и модели зафиксированы, рантайм-виджеты открыты)

---

## 0. Что это и откуда

Цель — браузерный **тренажёр на покрытие механизмов** платформы, тренирующий навык
**чтения и декомпозиции** экзаменационной задачи 1С:Специалист, а не зубрёжку готовых
решений.

Источник таксономии (не контента!) — сборник задач фирмы «1С» для подготовки к экзамену
по платформе 8.3 (версия экзамена сентябрь 2022). Параметры корпуса, подтверждённые
разбором:

- **167 задач** в 5 разделах: оперативный учёт (50), бухгалтерский (40), сложные
  периодические расчёты (50), бизнес-процессы (10), управляемые формы (17).
- Раздел 1: из ~600 уникальных предложений дословно общих — 2, встречаются ровно один
  раз — 500. То есть автор уже прогнал комбинаторику руками, каждый раз переформулируя.

Сборник используется **только как карта механизмов**. Текст заданий в продукте —
полностью наш, генерируемый (см. §5). Авторские формулировки не воспроизводятся.

---

## 1. Тезис: корпус параметричен

Каждая задача = `базовый_сценарий (каркас) + вектор значений по осям-механизмам`.

Доказательство на тексте:
- Задачи 1.1/1.2/1.3 имеют дословно совпадающий каркас (оптовая торговля, Приход/Расход,
  товары+услуги в одной ТЧ, склад не ведётся, контроль нехватки, отчёты), а различаются
  одним сменным модулем (партии по учётной политике → приоритет партии из шапки →
  построчный выбор партии).
- Задачи 3.1/3.2/3.3 — каркас «оклад по отклонениям + премия + командировка + отчёт»
  постоянен; крутятся независимые оси: единица времени (часы/дни), правило изменения
  оклада, премия (фикс % / история % / от стажа), база командировки, задним числом
  (да/нет), перерасчёты (вкл/выкл), вид отчёта.

**Вывод.** Единица аннотирования — *механизм*, а не *задача*. По частоте концептов
в учётных задачах: оплата/взаиморасчёты ≈72, партии ≈66, склад ≈64, граница
последовательности ≈29, цены ≈25, комплектация ≈14, далее длинный хвост (серии,
возвраты, резерв, скидки…). Оценка размера библиотеки — **~40–60 механизмов** на всю
книгу, каждый сидит в десятках задач.

## 2. Что строим (и чего не строим)

- **Строим:** тренажёр, гарантированно прогоняющий ученика по каждому механизму и по
  нетривиальным *комбинациям* механизмов.
- **Не строим:** «генератор бесконечных формулировок». Бесконечность — ложная метрика:
  оси не ортогональны, многие комбинации вырождаются или противоречивы. Ценность —
  в *покрытии* пространства решений, а не в объёме.

**Два слоя библиотеки (разной величины):**
| Слой | Размер | Происхождение | Кто верифицирует |
|---|---|---|---|
| Механизм (модель + эталон) | ~50, конечен | размечаем один раз | человек (автор), 1×/механизм |
| Поверхностный текст | неограничен | генерируем | не требует верификации (следует из модели) |

Разделение этих слоёв одновременно решает проблему авторских прав: книга даёт *механизмы*,
текст пишем сами.

---

## 3. Декомпозиция = уровни глубины (а не «сложность вообще»)

Эксперт решает задачу детерминированным конвейером декомпозиции. Его этапы = наши уровни:

| Уровень | Вопрос | Что ученик делает |
|---|---|---|
| **1. Чтение / классификация** | «что есть что» | относит сущности к видам объектов 1С |
| **2. Структура / реквизиты** | «из чего объект» | измерения/ресурсы/реквизиты/ТЧ |
| **3. Движения / алгоритм** | «что куда пишет и как» | движения + алгоритм проведения/расчёта |

**Ключ:** уровни — это *глубина декомпозиции одной и той же модели*, а оси-механизмы —
*ветвление на каждой глубине*. Не две сущности, а одна структура в двух срезах. Отсюда
режимы тренировки:
- **поперёк** — один уровень, много механизмов (набиваем распознавание вширь);
- **вдоль** — один механизм, все три уровня (копаем вглубь);
- **по диагонали** — 2–3 совместимых механизма в одной синтетической задаче (здесь
  оживает ограниченная комбинаторика, см. §9).

---

## 4. Градиент детерминированности = градиент проверяемости

Сложность растёт вместе с *неоднозначностью проверки*:

| Уровень | Режим проверки | Авто? |
|---|---|---|
| 1 | точное совпадение множества `{сущность → вид}` | да, мгновенно |
| 2 | совпало обязательное ядро (лишнее терпим) | да, почти полностью |
| 3 (декл.) | структурное сравнение со спеком движений | да |
| 3 (код) | разбор в AST + семантические ассерты | да, через BSLexicon |
| 3 (поведение) | прогон на тест-данных | через пошаговый симулятор |

**Следствие для MVP.** Самая широкая авто-проверяемая зона — у регистров накопления и на
уровнях 1–2. Это и есть «учим читать сначала». **MVP = РН + уровни 1–2.**

---

## 5. Две петли верификации (их нельзя путать)

- **Авторская верификация** — человек подтверждает *модель* карточки. Один раз на механизм.
- **Рантайм-проверка** — движок сверяет ответ ученика с *проекцией* модели. На каждой
  попытке, автоматически.

Поэтому **пред-генерация vs генерация на лету — это вопрос рендера, не корректности**:
любой сгенерённый вариант корректен, т.к. эталон — проекция той же выверенной модели.

**Решение по генерации:** фиксируем конечный набор выверенных *структур* (механизмы/комбо),
а *поверхность* (номенклатура, периоды, числа) рандомим на лету. Малый фиксированный N
без рандомизации поверхности → ученик запоминает N ответов, а не механизм.

## 6. Карточка механизма

### 6.1 Общий скелет (диспетчер) — ЗАФИКСИРОВАН

```yaml
MechanismCard:
  id: str                       # напр. partii_fifo_lifo
  kind: РН | РС | РБ | РР | БП | Форма     # диспетчер; задаёт форму поля model
  раздел: оперативный | бухгалтерский | расчёт | бизнес-процессы | формы
  навык: str                    # что тренирует, одной фразой
  контекст_требуется:           # предикат применимости (на каркас)
    - "оптовая торговля; Приход/Расход"
    - "склад не ведётся"
  вносит:                       # дельта, добавляемая в каркас при сборке
    объекты: [...]
    изменения: [...]
  ловушки:                      # педагогическое ядро уровня 1
    - соблазн: "Партия → Справочник"
      верно:   "измерение РН по регистратору; отдельного справочника нет"
      почему:  "партия идентифицируется документом-приходом"
  model: <тело, форма зависит от kind — см. §8>
  уровни:
    "1": { проекция: классификация, grade: точное_множество }
    "2": { проекция: структура,     grade: обязательное_ядро }
    "3": { проекция: движения,      grade: эталон, режимы: [декл, код, поведение] }
  совместимость:
    исключения:                 # основное выводится из контекста+вносит (см. §9)
      - { с: <id>, статус: конфликт | требует, причина: str }
```

### 6.2 `model` — типизированное объединение по `kind`

Все шесть kind делятся на **две метаформы**:

- **Структурные** (РН, РС, РБ, РР, БП): `model` = типизированный *граф объектов*
  (с рёбрами для РР и БП). Уровни 1–2 проверяются структурно.
- **Поведенческая** (Форма): `model` = спецификация событий/обработчиков + ограничения
  контекста компиляции. Проверка преимущественно кодовая/исполнительная; уровень 1 всё
  равно есть (распознавание событий и контекста).

### 6.3 Один источник истины

`model` — единственный источник истины. Уровни 1/2/3 — *проекции* (запросы) к ней,
а не отдельные данные. Размечаем модель один раз; уровни получаем проекцией. Это снижает
объём разметки и гарантирует согласованность уровней.

---

## 7. Движение — один семантический спек, много чекеров

Движение/поведение хранится как **один семантический спек**; «конструктор движений», код
и исполнение — это разные *чекеры над одним спеком*, а не разные данные. Иначе эталоны
разъедутся.

```
семантический спек движения  =  (регистр, вид, ресурс = выражение_от_источника)
        │
        ├── декларативно   → структурное сравнение (конструктор движений 1С)
        ├── код            → BSLexicon разбирает в AST → семантические ассерты
        └── поведение      → пошаговый симулятор прогоняет на тест-данных
```

BSLexicon — слой разбора BSL; семантический чекер строится поверх него. Где-то достаточно
конструктора движений, где-то нужен код — карточка несёт оба режима для одного спека.

## 8. Модели по семействам (полная раскладка)

Для каждого kind: объекты-партнёры · спец-структура · движение/поведение · оси-механизмы ·
где ломается плоская форма · зона авто-проверки.

### 8.1 РН — регистр накопления (раздел: оперативный)

```yaml
model(РН):
  объекты_партнёры: [ Перечисление?, РегистрСведений? ]
  регистр:
    подвид: остатки | обороты
    измерения: [ {имя, тип} ]        # Партия: ДокументСсылка.ПриходнаяНакладная
    ресурсы:   [ {имя} ]             # Количество, Стоимость
    реквизиты: [ ... ]
  движения:
    - документ: Приход
      записи: [ {вид: Приход, ресурсы: {Количество: "...", Стоимость: "..."}} ]
    - документ: Расход
      записи: [ {вид: Расход, алгоритм_id: списание_по_партиям} ]
```
- Оси: партии (FIFO/LIFO/шапка/построчно), склад, резерв, серии, обороты-vs-остатки,
  граница последовательности (задним числом).
- Плоская форма работает практически как есть.
- **Зона авто-проверки: широкая** (вся 1–2, большая часть 3-декл.). → MVP здесь.

### 8.2 РС — регистр сведений (разделы: оперативный/бухгалтерский)

```yaml
model(РС):
  регистр:
    подвид: периодический(период) | непериодический
    режим_записи: независимый | подчинён_регистратору
    измерения: [...]; ресурсы: [...]; реквизиты: [...]
    ведущие_измерения: [...]
  движение: запись   # для подчинённого — из документа
```
- Оси: учётная политика (период=год), история цен (период), графики, ведущие измерения.
- Спец-точка: *периодичность* и *срез* (последнее/первое/на дату) — отдельный навык.
- **Зона авто-проверки: широкая.**

### 8.3 РБ — регистр бухгалтерии (раздел: бухгалтерский)

```yaml
model(РБ):
  объекты_партнёры: [ ПланСчетов, ПланВидовХарактеристик(субконто) ]
  план_счетов:
    счета: [ {имя, предопределённый: bool, признаки_учёта: [Кол, Вал],
              забалансовый: bool, виды_субконто: [<ссылка на ПВХ>] } ]
  регистр:
    субконто: через_ПВХ            # переменная арность измерений, не плоский список
    ресурсы: [ Сумма, (Количество|Валюта по признаку учёта) ]
  движения:
    - документ: Расходная
      проводки: [ {Дт: Покупатели(субконто: [Контрагент, Договор]),
                   Кт: ПрибылиУбытки, Сумма: "..."} ]
    забалансовые: односторонняя_проводка
```
- Оси: разрез субконто (договоры, детализация до документа), валютный/количественный учёт,
  балансовый/забалансовый.
- **Где ломается плоское:** движение ≠ дельта ресурса, а пара Дт/Кт с субконто по сторонам;
  «измерения» = субконто переменной арности через ПВХ; новые объекты (План счетов, ПВХ);
  признаки учёта делают ресурсы условными.
- Новые ловушки L1: «счёт → справочник» (нет, План счетов); «субконто → измерение напрямую»
  (нет, ПВХ + настройка видов субконто).
- **Зона авто-проверки: средняя** (1–2 структурно, проводки структурно; забаланс — нюансы).

### 8.4 РР — регистр расчёта (раздел: сложные периодические расчёты)

```yaml
model(РР):
  объекты_партнёры: [ ПланВидовРасчёта, ГрафикРаботы(РС), Календарь, Перерасчёт ]
  ПВР:
    виды: [ Оклад, Премия, Командировка ]
    рёбра:                          # СУТЬ механизма живёт в рёбрах, не в списке
      вытеснение: [ {вид: Командировка, вытесняет: [Оклад, Премия]} ]
      база:       [ {вид: Командировка, по_видам: [Оклад, Премия], базовый_период: "2 мес"} ]
      ведущие:    [ ... ]
  регистр_расчёта:
    периодичность: Месяц
    график: ГрафикРаботы
    измерения: [...]; ресурсы: [ Результат, ... ]
  движение: запись { период_действия, период_регистрации }   # ≠ дельта ресурса
  расчёт:                           # ОТДЕЛЬНАЯ фаза
    источники: [ ДанныеГрафика, БазаРегистраРасчёта, ФактическийПериодДействия ]
    алгоритм_id: ...
  перерасчёт: { объект: Перерасчёт, триггер: [задним_числом, изменение_базы] }
```
- Оси: метод (отклонения), вытеснение по периоду действия, база за период, перерасчёт
  (тумблер), задним числом, стаж/прошлые периоды, часы↔дни.
- **Где ломается плоское сильнее всего:** ПВР — это *граф зависимостей* (вытеснение/база/
  ведущие — рёбра); запись движения и *расчёт* — две разные фазы; расчёт идёт над
  виртуальными таблицами; вытеснение и перерасчёт — *временно́е поведение*, не структура.
- L2 для РР — это **редактор графа** (зависимости видов расчёта), а не заполнение палитры.
- **Зона авто-проверки: узкая на L3.** Уровни 1–2 (это ПВР? это РР? вид вытесняющий?
  где база?) детерминированы; корректность расчёта/вытеснения/перерасчёта — поведенческая,
  упирается в исполнение (симулятор). Здесь симулятор обязателен.

### 8.5 БП — бизнес-процессы (раздел: бизнес-процессы)

```yaml
model(БП):
  объекты: [ БизнесПроцесс, Задача ]
  карта_маршрута:
    точки: [ Старт, Действие, Условие, ВыборВарианта, Разделение, Слияние,
             ВложенныйБП, Обработка, Завершение ]
    рёбра: [ {из, в, условие?} ]   # граф маршрута
  адресация:
    реквизиты_адресации: [...]     # часто через РС адресации / измерения адресации
  движение_аналог: маршрутизация (создание задач по точкам)
```
- Раздел маленький (10 задач) и слабо переиспользуемый по тексту → гранулярность механизмов
  крупнее, карточки ближе к «штучным».
- Спец-структура: граф маршрута + адресация задач.
- **Зона авто-проверки: средняя** (структура карты — да; маршрутизация — частично поведение).

### 8.6 Форма — управляемые формы (раздел: управляемые формы)

```yaml
model(Форма):
  мета_форма: поведенческая
  реквизиты_формы: [...]           # реквизит формы vs реквизит объекта — отдельная ловушка
  события: [ {имя: ПриИзмененииРеквизита, контекст: Клиент, эффект: "..."} ]
  контекст_компиляции: [ Клиент, Сервер, СерверБезКонтекста ]
  приёмы: [ условное_оформление, ОписаниеОповещения, программное_изменение_формы,
            динамический_список ]
  проверка: { L1: распознавание_событий_и_контекста, L3: код(BSLexicon)+исполнение }
```
- Раздел 5 (17 задач), высокая вариативность текста, нет общего каркаса.
- **Где ломается граф-модель:** это не граф объектов, а событийная модель + ограничения
  контекста клиент/сервер. Уровень 1 всё равно осмыслен (какое событие? какой контекст
  компиляции? реквизит формы или объекта?).
- **Зона авто-проверки: иная** — узкая на структуре, основная — код/исполнение.

---

## 9. Матрица совместимости — преимущественно вычисляется

Совместимость двух карточек **выводится**, не размечается целиком:
- совместимы, если `контекст_требуется` непротиворечивы И `вносит` не конфликтуют по объектам;
- руками задаём только редкие исключения (`совместимость.исключения`).

Это даёт сборку диагональных задач (§3) как *счётную* операцию, при этом карточка остаётся
атомарной.

## 10. Зафиксированные решения (с обоснованием)

1. **Единица — механизм, не задача.** Корпус параметричен (§1, данные).
2. **Тренажёр на покрытие механизмов, не генератор формулировок** (§2).
3. **Уровни = глубина одной модели; оси = ветвление** (§3). Срезы вдоль/поперёк/диагональ.
4. **Градиент сложности = градиент проверяемости; MVP = РН + L1–L2** (§4).
5. **Две петли верификации; пред-ген vs на-лету = только рендер** (§5). Фиксируем структуры,
   рандомим поверхность.
6. **Карточка: один `model` → проекции уровней** (§6.3).
7. **Диспетчер `kind` + общий скелет зафиксированы; тело `model` типизировано по kind** (§6, §8).
8. **Движение — один спек, чекеры: декл./код(BSLexicon)/исполнение(симулятор)** (§7).
9. **Текст генерируется (обход авторских прав); сборник — только таксономия** (§0, §2).
10. **Матрица совместимости вычисляется из контекста+вносит** (§9).

## 11. Открытые вопросы

- **Рантайм-виджеты L1–L2:** drag-drop по корзинам vs ввод с проверкой по множеству vs
  предсказание с раскрытием; формат фидбэка (как раскрывать ловушку).
- **Граф-редактор для РР (ПВР) и БП (карта маршрута):** какой виджет на L2.
- **Форма:** оставляем ли полноценные уровни 1–2 или это преимущественно L3-кодовый kind.
- **Грань L3 декл. ↔ код:** для каких механизмов конструктора движений достаточно, а где
  обязателен BSL.
- **Стык с симулятором:** где проходит граница «уровень 3 структурный» → «уровень 4
  поведенческий (исполнение)».
- **Формализация эталона L3** для авто-сравнения с допустимыми вариациями.

## 12. Дорожная карта MVP

1. Закрыть скелет карточки (§6.1) как схему + 1 заполненный эталон: **партии FIFO/LIFO (РН)**.
2. Чекеры L1 (точное множество) и L2 (обязательное ядро).
3. Виджеты L1–L2 + фидбэк с раскрытием ловушек.
4. Рендер поверхности (рандомизация номенклатуры/периодов/чисел над фиксированной структурой).
5. 8–12 карточек РН → первый «поперёк»-режим (распознавание вширь).
6. Затем РС; РБ/РР/БП/Форма — по готовности их моделей и чекеров.
