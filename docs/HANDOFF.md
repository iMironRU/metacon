# МетаКон — HANDOFF для Claude Code

`v0.1.0` · бриф для сессии разработки MVP. Полное обоснование решений — в `docs/CONCEPT.md`.
Этот файл переводит концепцию в конкретные тех-решения и первые задачи. Дизайн **не**
переобсуждаем — он зафиксирован.

## Идентификация

- Имя: **МетаКон** (начертание МетаКон / MetaCon), репозиторий `iMironRU/metacon`.
- Стек: **Vite + React + TypeScript**, статическая сборка → **GitHub Pages**, **без бэкенда**.

## Инварианты архитектуры (не нарушать)

1. **Карточки = данные, движок = код, разделены.** Карточки — JSON по
   `schema/mechanism-card.schema.json`. Чекеры — чистые функции `(карточка, ответ) → вердикт`
   (`engine/contracts.ts`).
2. **Две петли верификации.** (а) Авторская — человек подтверждает модель карточки, 1×/механизм.
   (б) Рантайм — движок сверяет ответ ученика с проекцией модели, на каждой попытке.
   `*.golden.json` — приёмочный тест карточки: чекеры обязаны воспроизвести его verdict.
3. **Один `model` → проекции уровней.** Уровни 1/2/3 — это проекции одной модели, не отдельные данные.
4. **MVP детерминирован и безсерверный.** Грейдинг L1–L2 в браузере; карточки — статический JSON.
5. **Текст генерируется из `render`-пулов** (наш контент). Авторские формулировки книги не используются.

## Решение по именованию полей

Ключи схемы/кода — **латиницей** (контракт), значения предметной области — **на русском**
(`object_kind: "РегистрНакопления"`). В `docs/CONCEPT.md` ключи по-русски — это для читаемости
дизайн-документа; в коде источник истины — JSON-схема с латинскими ключами.

## Дерево репозитория (целевое)

```
metacon/
  docs/        CONCEPT.md (дизайн), HANDOFF.md (этот файл)
  schema/      mechanism-card.schema.json   ← контракт карточки (source of truth)
  cards/
    operativny/
      partii_fifo_lifo.json          ← seed-эталон (РН)
      partii_fifo_lifo.golden.json   ← приёмочные golden-ответы L1/L2
  engine/      contracts.ts (сигнатуры чекеров) + checkers (реализует Code)
  ui/          виджеты L1/L2 (реализует Code)
  .github/workflows/deploy.yml       ← Vite → Pages
  (package.json, vite.config.ts, tsconfig — генерит Code через `npm create vite`)
```

## Контракты

- **Источник истины** — JSON-схема. Типы данных: `npx json-schema-to-typescript schema/mechanism-card.schema.json -o engine/card.types.ts`.
- **Чекеры** реализуют интерфейсы из `engine/contracts.ts`:
  - L1 `точное_множество`: ответ `{сущность → корзина}` точно равен `expected`; иначе fail + какая ловушка сработала.
  - L2 `обязательное_ядро`: `expected_core.dimensions/resources` ⊆ ответа; лишнее терпим; нарушение роли (партия как ресурс) — fail.
- **Уровни — проекции `model`:** L1 классификация объектов; L2 структура `register`; L3 (вне MVP) — движения.

## Первые задачи (порядок)

1. `npm create vite@latest` (react-ts), подключить `base: '/metacon/'` в `vite.config.ts`.
2. Кодген типов из схемы → `engine/card.types.ts`. Валидация карточек по схеме (ajv) в тесте.
3. Реализовать `l1Checker` и `l2Checker` (`engine/checkers.ts`) против `engine/contracts.ts`.
4. **Прогнать golden-фикстуры как тест** (`partii_fifo_lifo.golden.json`) — это приёмка эталона.
5. Виджет L1: drag-drop сущностей по корзинам (+ ловушки-дистракторы), мгновенный фидбэк, раскрытие ловушки.
6. Виджет L2: сбор измерений/ресурсов из палитры, проверка по ядру.
7. Рендер поверхности: подстановка из `render`-пулов поверх фиксированной модели.
8. Загрузка seed-карточки, прохождение L1→L2 end-to-end.
9. Деплой через `.github/workflows/deploy.yml`.

## Что ВНЕ MVP (не делать сейчас)

- Уровень 3 (движения/код/исполнение), интеграция BSLexicon и пошагового симулятора.
- Семейства РС/РБ/РР/БП/Форма (тело `model` — заглушка в схеме; CONCEPT §8 — справка).
- Матрица совместимости и диагональная сборка (CONCEPT §9).
- Генерация на лету сверх рандомизации поверхности; бэкенд; аккаунты.

## Definition of done (MVP)

Ученик открывает задачу по `partii_fifo_lifo`, проходит уровень 1 (классификация) и
уровень 2 (структура регистра) с детерминированной проверкой и фидбэком по ловушкам;
golden-фикстуры зелёные; статика задеплоена на Pages.
